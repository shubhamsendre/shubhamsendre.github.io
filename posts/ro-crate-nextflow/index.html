What Is RO-Crate?

RO-Crate (Research Object Crate) is a community standard for packaging research outputs — workflows, data, code, results — together with structured metadata that makes them self-describing.

The idea is simple: a folder of pipeline outputs means nothing to anyone except the person who ran it. An RO-Crate is that same folder with a ro-crate-metadata.json at the root that answers the questions any reader would ask:

  • What is this?
  • Who created it?
  • What inputs were used?
  • What tools ran, at what versions?
  • What outputs were produced?
  • When did it run?

The metadata uses schema.org vocabulary — the same structured data standard that underpins Google’s knowledge graph — which means it is both human-readable and machine-parseable by any system that understands the spec.

RO-Crate is not a new file format. It is a convention: any directory with a ro-crate-metadata.json that conforms to the RO-Crate specification is a valid RO-Crate.

Two Levels of RO-Crate in a Nextflow Pipeline

Before going further, it helps to understand that RO-Crate applies at two distinct levels in a Nextflow context:

Level What it describes When it exists
Pipeline-level The workflow itself — purpose, authors, license, inputs, outputs Once per pipeline, committed to the repo
Run-level A specific execution — input files, output files, parameters, container versions, timestamps Generated fresh per pipeline run

Both are valid RO-Crates. They serve different purposes. The pipeline-level one makes your workflow a citable, discoverable artifact. The run-level one is your audit trail and reproducibility record.

Pipeline-Level RO-Crate: Describing Your Workflow

A pipeline-level ro-crate-metadata.json lives in the root of your pipeline repository. It describes your workflow as a ComputationalWorkflow — a scientific artifact with a name, authors, license, and formal descriptions of its inputs and outputs.

The Minimal Structure

Every valid RO-Crate metadata file has two required entries in its @graph:

{
  "@context": "https://w3id.org/ro/crate/1.1/context",
  "@graph": [
    {
      "@id": "ro-crate-metadata.json",
      "@type": "CreativeWork",
      "conformsTo": {"@id": "https://w3id.org/ro/crate/1.1"},
      "about": {"@id": "./"}
    },
    {
      "@id": "./",
      "@type": ["Dataset", "ComputationalWorkflow"],
      "name": "my-annotation-pipeline",
      "description": "A Nextflow pipeline for variant annotation using VEP, FastQC, and MultiQC.",
      "license": "MIT",
      "version": "1.0.0",
      "programmingLanguage": {
        "@id": "https://w3id.org/workflowhub/workflow-ro-crate#nextflow"
      },
      "author": [{"@id": "#author-1"}],
      "input": [{"@id": "#input-samplesheet"}],
      "output": [{"@id": "#output-vcf"}]
    }
  ]
}

The first entry (ro-crate-metadata.json) is the metadata descriptor — it always points to ./ as its subject. The second entry (./) is the root dataset — your pipeline.

Adding Authors

{
  "@id": "#author-1",
  "@type": "Person",
  "name": "Shubham Sendre",
  "affiliation": "Invisibl Cloud Solutions Pvt. Ltd.",
  "url": "https://linkedin.com/in/shubhamsendre"
}

If you have an ORCID, use that as the @id instead of a local fragment — it makes the pipeline citable in academic contexts.

Describing Inputs and Outputs

Formally documenting inputs and outputs using FormalParameter is what makes the pipeline machine-readable. Tools like WorkflowHub parse these to display pipeline interfaces.

{
  "@id": "#input-samplesheet",
  "@type": "FormalParameter",
  "name": "input",
  "description": "Samplesheet CSV with sample IDs and paths to FASTQ files",
  "encodingFormat": "text/csv"
},
{
  "@id": "#output-vcf",
  "@type": "FormalParameter",
  "name": "annotated_vcf",
  "description": "VEP-annotated VCF with functional consequence predictions",
  "encodingFormat": "text/tab-separated-values"
}

Referencing Software Dependencies

Use bio.tools identifiers where they exist — they are the standard registry for bioinformatics tools.

{
  "@id": "https://bio.tools/vep",
  "@type": "SoftwareApplication",
  "name": "Ensembl VEP",
  "version": "110.1",
  "url": "https://www.ensembl.org/vep"
}

Add this to hasPart or softwareRequirements on the root Dataset node.

Run-Level RO-Crate: Provenance with nf-prov

The pipeline-level RO-Crate describes your workflow as a static artifact. Run-level provenance captures what actually happened in a specific execution — which input files, which container versions, which parameters, what outputs, and when.

The nf-prov plugin handles this in Nextflow.

Enabling nf-prov

Add to your nextflow.config:

plugins {
  id 'nf-prov@1.2.2'
}

prov {
  enabled = true
  formats {
    rocrate {
      file = "${params.outdir}/pipeline_info/ro-crate-metadata.json"
    }
  }
}

What Gets Captured Per Run

After every pipeline run, nf-prov writes an ro-crate-metadata.json into your output directory that records:

  • Workflow version (exact git commit or tag)
  • Input files with checksums (MD5/SHA256)
  • Output files with checksums and sizes
  • Start and end timestamps per process
  • Container image IDs used per process
  • Full resolved parameter set at runtime

A fragment of what this looks like:

{
  "@id": "#run-2026-04-15",
  "@type": "CreateAction",
  "instrument": {"@id": "main.nf"},
  "startTime": "2026-04-15T09:23:11Z",
  "endTime": "2026-04-15T11:47:32Z",
  "object": [{"@id": "inputs/samplesheet.csv"}],
  "result": [{"@id": "results/annotated.vcf.gz"}]
}

This is the difference between “I ran the pipeline” and “I can prove exactly what ran, on what input, at what time, and what came out.”

Publishing to WorkflowHub

WorkflowHub is the community registry for FAIR scientific workflows. It reads your pipeline-level ro-crate-metadata.json directly to:

  • Display pipeline metadata, authors, description, and inputs/outputs
  • Assign a DOI to your pipeline version, making it citable in publications
  • Link to bio.tools and Bioschemas entries for referenced software
  • Index your pipeline for discoverability across the research community

Registering is straightforward: you provide a link to your repository or upload a zip, and WorkflowHub validates your RO-Crate and creates a landing page.

A DOI-backed WorkflowHub entry is also a portfolio item — a versioned, citable artifact with your name on it that you can link from your CV or LinkedIn.

Why This Matters Beyond Good Practice

Three concrete reasons to invest time in RO-Crate now:

Reproducibility is becoming a requirement. Journals, funders, and institutions increasingly require that computational analyses be reproducible. An nf-prov generated RO-Crate is concrete evidence — not just “here’s the code,” but “here’s the exact execution with checksums.”

Client and audit requirements. Delivering pipeline outputs to external clients — pharma companies, hospitals, research institutes — often comes with a question: can you prove what ran? An RO-Crate run record is a ready-made answer.

The bridge to regulated environments. Trusted Research Environments (TREs) — secure compute environments for sensitive clinical data used in NHS, UK Biobank, and EU health data spaces — require data provenance and audit trails by design. RO-Crate is one of the standards being evaluated and adopted to satisfy those requirements.
